Privacy Policy

Effective date: June 20, 2026
Version: Aurora (1.0)
License: GNU General Public License v3.0 (GPLv3)
Source code: https://github.com/r1nzd/KydeOS

1. Overview

KydeOS is a free, open-source operating system designed and maintained by a single independent developer. This Privacy Policy explains what data KydeOS may collect, how it is used, and what control you have over it.

Privacy in KydeOS is opt-in by design. During the initial setup process (OOBE), you are presented with a Privacy & Diagnostics screen where you can individually enable or disable each data collection feature before the system is fully active. Every toggle defaults to off, with the exception of Crash Reports which defaults to on to help identify critical bugs during early development. You can change any of these settings at any time after setup via System Settings.

KydeOS has no advertising business model, no investors requiring growth metrics, and no commercial interest in your data. This project exists as a personal, open-source endeavor.

2. Privacy & Diagnostics Settings

During setup, you are given direct control over the following three settings. None of these features transmit any data without your knowledge.

2.1 Location Services

Default state: OFF

When enabled, Location Services allows KydeOS and supported applications to access an estimated location derived from your network or hardware. This is used to assist apps that benefit from location context, such as weather, maps, and automatic time zone detection.

When disabled, no location data of any kind is collected, accessed, or transmitted by the system. Individual applications you install may have their own location permission requests, which are handled separately and are not governed by this system-level toggle.

You can change this setting at any time in System Settings.

2.2 Telemetry & Usage Data

Default state: OFF

When enabled, KydeOS may collect anonymous, aggregated data about how the system is used. This may include general information such as which system features are accessed, session durations, or performance metrics. The purpose of this data is strictly to understand how KydeOS is used in practice and to guide future development decisions.

This data is:

  • Anonymous — it is not linked to your identity, account, or any personally identifiable information
  • Aggregated — individual sessions are not stored or analyzed in isolation
  • Never sold or shared with any third party
  • Never used for advertising purposes of any kind

When disabled, no usage data or telemetry of any kind is collected or transmitted by the system.

You can change this setting at any time in System Settings.

2.3 Crash Reports

Default state: ON

When enabled, KydeOS automatically collects and transmits error logs when the system or a core system component crashes. These reports contain technical diagnostic information such as stack traces, system state at the time of the crash, kernel version, and hardware architecture. Crash reports help the developer identify and fix critical bugs, especially during the early Aurora development phase.

Crash reports do not contain:

  • Your name, email address, or any account information
  • The contents of your files, documents, or clipboard
  • Your browsing history or application usage
  • Any personally identifiable information

When disabled, no crash or error data is ever transmitted from your device.

You can change this setting at any time in System Settings.

3. Data We Never Collect

Regardless of your Privacy & Diagnostics settings, KydeOS never collects the following under any circumstances:

  • Names, email addresses, or personally identifiable information during normal OS operation
  • The contents of your files, documents, photos, or any stored data
  • Keyboard input or clipboard contents
  • Browsing history or Krowser activity
  • Hardware serial numbers or unique device identifiers used for tracking
  • Location data when Location Services is disabled
  • Any data sold to or shared with advertisers, data brokers, or commercial third parties

4. Krowser

Krowser is the first-party browser bundled with KydeOS. When you browse the web, Krowser connects to the websites you visit — those websites operate under their own privacy policies. Krowser does not report your browsing history, visited URLs, or search queries back to the KydeOS developer in any form, regardless of your Privacy & Diagnostics settings.

5. KydeStore & Developer Program

KydeStore is the native software center in KydeOS. An alpha developer program is available, allowing early community members to register as testers directly inside KydeStore. If you register, the information you voluntarily submit is used solely for managing the program and is never shared with third parties.

6. Third-Party Applications

Applications installed via .deb, Flatpak, or AppImage are independent software with their own developers and privacy practices. KydeOS has no control over or responsibility for the data practices of third-party applications. Consult each application's privacy policy individually.

7. System Update Checks

When KydeOS checks for available updates, a standard network request is made to the update server. Your IP address is exposed to that server as an unavoidable consequence of internet networking. No additional data is transmitted alongside this request.

8. Open Source as a Privacy Guarantee

KydeOS is fully open source under the GPLv3. Every component of the system is publicly auditable at https://github.com/r1nzd/KydeOS. You are never required to take our word for anything stated in this policy — you can verify it directly in the source code. If you discover any undisclosed data collection or network transmission in KydeOS, please report it immediately as a privacy issue on GitHub.

9. Security

KydeOS includes the following system-level security features:

  • LUKS2 full-disk encryption to protect data stored on your device at rest
  • AppArmor and SELinux sandboxing to isolate applications
  • Minimal default service configuration to reduce attack surface
  • Security patches from Debian Stable upstream, with critical fixes applied out-of-cycle when necessary

10. Your Rights

If you participated in the KydeStore developer program or any associated service, you have the right to request access to, correction of, or deletion of any personal data held about you. Contact the developer via GitHub at https://github.com/r1nzd/KydeOS.

11. Children's Privacy

KydeOS does not knowingly collect personal data from children under the age of 13. If you believe a minor has submitted personal information through KydeStore or any associated service, contact the developer via GitHub so it can be promptly removed.

12. Changes to This Policy

All changes to this policy will be committed to the public GitHub repository with a full auditable history. The effective date at the top of this document will be updated with each revision. Continued use of KydeOS after changes are published constitutes acceptance of the updated policy.

What is KydeOS?

Most Linux distributions fall into one of two camps. On one side you have the polished, user-friendly desktops that ship with years of accumulated bloat, questionable default choices, and telemetry you never asked for. On the other side you have the minimal, highly configurable systems that demand significant time investment before they become livable. KydeOS is an attempt to occupy the space in between — a system that boots fast, stays out of your way, looks genuinely good, and gives you the full power of Debian without making you fight for it.

The design language is Material 3 Expressive, applied consistently across every surface of the desktop. The shell is custom-built in Rust. The browser is first-party. The software center is native. Nothing is bolted on as an afterthought.

This is a solo project built in spare time. It is open source not as a marketing strategy but because it should be. Every line of code is public, auditable, and yours to study, modify, and redistribute under the GPLv3.

Key Features

Material 3 Expressive UI

The entire desktop experience is built around Google's Material 3 design language in its Expressive variant. This means dynamic color palettes that adapt to system context, fluid surface elevation, shape morphing across component groups, and consistent accessible typography throughout. Window decorations, system dialogs, notifications, the app drawer, the settings panel — everything follows the same design system with no exceptions. There is no mixing of design languages, no legacy GTK widgets slipping through the cracks, no visual inconsistency between system components.

KydeShell

KydeShell is the custom desktop shell developed specifically for KydeOS, written from scratch in Rust. It is not a fork of an existing shell and does not wrap any pre-existing desktop environment. KydeShell handles window management, the taskbar, the system tray, notifications, the application launcher, and workspace management. Being built in Rust means it is memory-safe, fast to start, and has a minimal runtime footprint. KydeShell is deeply integrated with the KydeOS environment in ways that a generic shell cannot be.

Krowser

Krowser is a first-party web browser developed by the KydeOS developer and bundled as the default browsing experience. It is designed to fit visually and functionally within the KydeOS environment rather than being a generic browser dropped in as a placeholder. Krowser does not report browsing history, visited URLs, or usage data back to the developer in any form.

KydeStore

KydeStore is the native software center built into KydeOS. It serves as the primary interface for discovering, installing, and managing applications, themes, and tools curated for the KydeOS environment. KydeStore is not a wrapper around an existing package manager frontend — it is a purpose-built application designed to feel native to the system. An alpha developer program is available, allowing early community members to register as testers directly inside KydeStore.

No bloat, no Snap, no telemetry

KydeOS ships with only what is essential. No Snap packages. No unnecessary background daemons. No opt-out telemetry buried in settings. Privacy & Diagnostics options are presented clearly during the initial setup process and default to off. You choose what the system sends, if anything, before the system is fully active.

Debian Bookworm foundation

KydeOS is built directly on top of Debian Bookworm Stable. This means access to one of the largest, most battle-tested APT package repositories in the Linux ecosystem, long-term security update cadence, excellent hardware support via the Linux LTS kernel, and a proven base that has been in production use across servers and desktops worldwide for decades. Debian's stability is not incidental to KydeOS — it is the foundation everything else is built on.

Flatpak and AppImage support

In addition to native .deb package installation, KydeOS supports Flatpak and AppImage out of the box. Snap is explicitly not supported and is not installed. You install software the way you want to without being forced into a single distribution model.

Wayland-first display

KydeOS uses Wayland as its primary display server. XWayland is available as a fallback layer for legacy applications that have not yet been ported to Wayland-native rendering. This means lower input latency, better display scaling support, and improved security isolation between applications compared to a pure X11 stack.

Full-disk encryption

LUKS2 full-disk encryption is supported out of the box and can be configured during the initial setup process. AppArmor and SELinux sandboxing provide additional isolation at the application level. The default service configuration is intentionally minimal to reduce attack surface.

Broad localization

KydeOS ships with support for 90 or more languages, configurable during the out-of-box experience.

Dual architecture support

KydeOS targets both x86_64 and ARM architectures. Both UEFI and Legacy BIOS boot targets are supported.

Build & Development

KydeOS is built inside a WSL2 Debian environment running on Windows 10. The root filesystem is manipulated directly at the lowest level using live-build and debootstrap. No graphical remaster tools are used at any point in the build process.

Build environment:

  • Host: WSL2 (Debian), username ace, hostname Admin
  • Tools: live-build, debootstrap, mksquashfs, custom shell scripts
  • Testing: QEMU with OVMF firmware for UEFI boot validation
  • CI/CD: GitHub Actions for automated build, compression, and release management

Build process:

  • Step 1 — Rootfs construction. The base Debian Bookworm root filesystem is bootstrapped using debootstrap and configured using live-build. System packages, configuration files, custom scripts, and KydeOS-specific components are injected directly via CLI.
  • Step 2 — Integrity verification. Before packaging, the rootfs is manually inspected for broken symlinks, missing dependencies, misconfigured permission bits, and any anomalies in critical system directories.
  • Step 3 — Packaging. The verified rootfs is compressed using mksquashfs and assembled into a bootable ISO filesystem structure via lb build. WSL2 memory allocation is configured via .wslconfig to prevent mksquashfs crashes during compression.
  • Step 4 — Virtual machine testing. The packaged ISO is booted inside QEMU with OVMF UEFI firmware. The kernel, initrd, storage controllers, network interfaces, and display outputs are all validated. Memory usage, CPU load, and disk I/O are profiled to catch performance regressions.
  • Step 5 — CI/CD pipeline. On push to the main branch, GitHub Actions runners execute the full build sequence automatically, output a bootable image, draft release tags, and run security scans for compromised dependencies or outdated certificates.

Release Channels

Nightly

Automated builds generated from the latest commit on the main branch. These builds may be broken, incomplete, or contain regressions. Intended exclusively for developers and testers who want to track the absolute latest state of the codebase.

Alpha

Early feature builds produced during active development phases. Core functionality is present but stability is not guaranteed. Intended for early adopters and developer program participants who are comfortable with instability and want to provide feedback.

Beta

Feature-complete builds in the bug-fixing and stabilization phase. Most functionality works as intended. Regressions are being actively addressed. Suitable for more adventurous users who want to test before the stable release.

Stable

Fully tested, hardware-validated releases. These builds have passed the complete six-month development cycle including physical hardware testing. Recommended for general use.

Release Cycle

KydeOS follows a strict six-month release cycle designed to maintain development sustainability and prevent burnout.

  • Month 1 — Rest and recovery. No new features are introduced. The developer rests completely. This is non-negotiable and exists to ensure long-term project health.
  • Months 2 and 3 — Research and experimentation. New upstream Debian packages are evaluated. Cutting-edge desktop components are tested. The Material 3 theme architecture is evolved. No production commits are made during this phase.
  • Months 4 and 5 — Active development and packaging. Core development work happens here. New features are integrated, bugs are fixed, and the rootfs configuration is solidified. This is the primary engineering phase of each cycle.
  • Month 6 — Finalization and deployment. Rigorous physical hardware testing across multiple configurations. Documentation is updated. The stable ISO is compiled, validated, and officially published.

Critical security patches are applied out-of-cycle at any time when necessary, regardless of which phase the project is currently in.

Development Roadmap

Phase 1 — Stable bootable base image.
Establish a completely stable, bootable KydeOS base image derived directly from Debian sources. The system should boot reliably in QEMU via OVMF with a functional TTY and no kernel panics or mounting failures.
Status: Complete. KydeOS currently boots to a functional TTY in QEMU.

Phase 2 — Material 3 UI/UX integration.
Complete deep integration of the custom Material 3 Expressive UI across all display managers and desktop surfaces. KydeShell feature-complete. All system applications styled consistently. Wayland compositor finalized.
Status: In progress.

Phase 3 — Automated nightly builds.
Implement fully automated nightly build generation via GitHub Actions to facilitate continuous public testing and hardware validation across x86_64 and ARM targets.
Status: Planned.

Phase 4 — Native app ecosystem and update infrastructure.
Deploy dedicated infrastructure for system updates and package distribution. KydeStore production-ready. Developer program launched publicly. Dedicated APT repository live. Full 90+ language localization validated.
Status: Planned.

Who Is KydeOS For?

Power users and enthusiasts

If you want Debian's stability with a desktop that looks and feels genuinely modern without spending days assembling it yourself, KydeOS is built for you. Everything is configured, themed, and ready from first boot.

Minimalist developers

A sub-10-second boot time, approximately 300 MB idle RAM, no unnecessary background services, and a clean Debian base underneath. A fast, distraction-free environment for people who spend most of their time in a terminal anyway.

Open source advocates

Every line of KydeOS is public, auditable, and freely redistributable under the GPLv3. If you care about software freedom and want to see or contribute to a completely transparent, independently built operating system, KydeOS is built in that spirit.
